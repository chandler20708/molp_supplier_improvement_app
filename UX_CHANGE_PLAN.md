# UX Change Plan

Goal: make the app help stakeholders quickly understand suppliers and act consistently with the final report recommendation.

## Phase 1: Fix Trust and Runtime Polish

### 1. Remove visible Plotly warnings

Problem: Streamlit displays Plotly deprecation warnings inside chart areas.

Change:

- Replace chart calls like `st.plotly_chart(fig, width="stretch")` with the current Streamlit-supported API using `config=`.
- Check all pages using Plotly charts:
  - `pages/1_Portfolio_Overview.py`
  - `pages/2_Supplier_Deep_Dive.py`
  - `pages/3_Scenario_Simulator.py`
  - `pages/4_Sensitivity_Export.py`

Acceptance check:

- No yellow deprecation warnings appear on any page.

### 2. Confirm deployment dependencies

Problem: the app failed until `plotly` was installed in the active Python environment.

Change:

- Keep `plotly` in `requirements.txt`.
- Add a short startup/deployment check to the README explaining that the app should be run from the app folder with dependencies installed.

Acceptance check:

- `pip install -r requirements.txt`
- `streamlit run app.py`
- All pages render.

## Phase 2: Make the App Open with the Recommendation

### 3. Replace landing page with Executive Summary

Problem: the current landing page shows project structure instead of supplier decisions.

Change `app.py` to show:

- Preferred strategic suppliers: H, B, A.
- Operational benchmarks: G, I.
- Primary development candidate: C.
- Conditional development candidates: F, K, D.
- Lower-priority suppliers: E, J, L.
- One sentence explaining the logic: MCDA quality/service + DEA efficiency + MOLP development potential.

Remove or move:

- "Expected project structure" code block.
- Deployment-folder explanation.

Acceptance check:

- A stakeholder can see the final recommendation without opening another page.

### 4. Rename the sidebar home label

Problem: sidebar label `app` is not meaningful.

Change:

- Rename home/navigation display to `Decision Summary` if Streamlit page naming allows it.
- If not, add a strong page title and avoid relying on the sidebar label.

Acceptance check:

- Navigation reads like a decision workflow, not a file structure.

## Phase 3: Align Supplier Classification with Report Logic

### 5. Add report recommendation tiers

Problem: automatic median-threshold portfolio statuses conflict with the report recommendation.

Change:

- Add a stable recommendation tier mapping, probably in `utils/recommendations.py` or a new `utils/recommendation_tiers.py`.

Suggested mapping:

```python
RECOMMENDATION_TIERS = {
    "H": "Preferred strategic supplier",
    "B": "Preferred strategic supplier",
    "A": "Preferred strategic supplier",
    "G": "Operational benchmark",
    "I": "Operational benchmark",
    "C": "Primary development candidate",
    "F": "Conditional development candidate",
    "K": "Conditional development candidate",
    "D": "Conditional development candidate",
    "E": "Lower-priority supplier",
    "J": "Lower-priority supplier",
    "L": "Lower-priority supplier",
}
```

Use this tier in:

- Executive Summary.
- Portfolio Overview table.
- Supplier Deep Dive header.
- Scenario Interpretation shortlist.

Acceptance check:

- I is not displayed simply as `Deprioritise`.
- F, K and D are visible as conditional development cases.
- L is lower priority, not a primary development candidate.

### 6. Keep analytical status as secondary

Problem: `portfolio_status` is still useful but should not override the final recommendation.

Change:

- Rename current `portfolio_status` in the UI to `Analytical status` or `MCDA-DEA status`.
- Show recommendation tier first.

Acceptance check:

- Users can distinguish model-derived classification from final management recommendation.

## Phase 4: Fix Supplier Deep Dive Action Logic

### 7. Make primary action scenario-bottleneck driven

Problem: primary actions are generated by fixed metric order, so price often appears first even when the scenario bottleneck is delivery or lead time.

Change:

- In `generate_recommendation_summary`, derive priority actions from `build_improvement_table` or from the same normalised gap logic used in `build_scenario_potential_table`.
- Exclude `Purchase` and `Customer Service Overlay` from MOLP action ranking.
- Select the largest normalised actionable gap as the primary action.

Acceptance check:

- Supplier C under Balanced: primary action is late delivery.
- Supplier F under Balanced: primary action is shipping errors.
- Supplier K under Balanced: primary action is lead time.
- Supplier D under Balanced: primary action is late delivery.

### 8. Add report-specific action notes

Problem: conditional supplier guidance is too generic.

Change:

- Add short supplier-specific management notes:
  - H: premium supplier.
  - B and A: scalable sourcing partners.
  - B: primary operational benchmark.
  - C: strongest development candidate; late delivery first.
  - F: only develop if shipping errors/service weakness can be reduced.
  - K: lead-time improvement case.
  - D: close to frontier but late-delivery constrained.
  - E, J, L: monitor, renegotiate or use selectively.

Acceptance check:

- Deep Dive for each supplier gives the same management interpretation as the final report.

## Phase 5: Improve Portfolio Comprehension

### 9. Add an action table before the 3D chart

Problem: the 3D chart is useful but not the fastest decision view.

Change:

- Add a table/cards section above the 3D matrix:
  - Supplier
  - Recommendation tier
  - Why it sits there
  - Immediate action
  - Main risk/caveat

Acceptance check:

- Users can understand the portfolio without interacting with the 3D chart.

### 10. Keep the 3D matrix as evidence, not the headline

Problem: 3D chart requires interpretation.

Change:

- Position it as "Evidence view" after the action table.
- Consider adding a simpler 2D matrix:
  - x-axis: strategic attractiveness or product quality.
  - y-axis: DEA efficiency or development potential.
  - color: recommendation tier.

Acceptance check:

- The first visible portfolio section is decision-oriented.

## Phase 6: Navigation and Language

### 11. Rename Scenario Simulator

Problem: "Simulator" overpromises because live MOLP is secondary/unavailable in many environments.

Change:

- Rename file/page label if practical from `Scenario Simulator` to `Scenario Interpretation`.
- Keep live MOLP as a secondary tab.

Acceptance check:

- Sidebar and page title use the same name.

### 12. Reduce method text above decisions

Problem: pages explain methods before giving answers.

Change:

- Put plain-language decision boxes first.
- Move detailed method notes into expanders or the Method Notes tab.

Acceptance check:

- Each page answers the management question in the first viewport.
